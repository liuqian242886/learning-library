OVERVIEW
课程概览
本课程解决 AI Agent 最核心的局限——会话间失忆。通过将记忆工程化为"一等公民"基础设施，让 Agent 真正实现跨会话持久学习。
💡
核心理念：大多数 Agent 每次会话结束后就"失忆"。本课程将记忆作为第一公民的基础设施——外置于模型、持久化存储、结构化管理，从根本上解决 Agent 的失忆问题。
CORE CONCEPT
四类记忆体系
课程将人类认知科学的记忆分类映射到 Agent 工程中，为设计持久化记忆系统提供清晰的理论框架。
⚡
Working Memory

工作记忆

当前 Context Window 中的内容——LLM 正在处理的对话和信息。有 Token 上限，会话结束即消失。

类比：人脑在计算时暂存的中间结果
📖
Episodic Memory

情节记忆

历次会话/事件的原始流水账，带时间戳。记录"发生了什么"，可按时间范围或语义查询。

类比：每日日记，记录具体发生的事情
🗂️
Semantic Memory

语义记忆

从情节记忆中提炼的结构化事实——用户偏好、领域知识、实体关系。持久存储，支持语义检索。

类比：百科全书，归纳后的通用知识
🔧
Procedural Memory

程序记忆

Agent 的行为规则、工作流、工具调用策略（System Prompt / Tool Descriptions）。指导"如何行动"。

类比：肌肉记忆，骑车无需思考
记忆类型 存储位置 持久化 检索方式 典型内容
工作记忆 Context Window（模型内） ✗ 不持久 直接读取 当前对话轮次
情节记忆 外部数据库（时序） ✓ 持久 时间范围 / 语义 历史会话日志
语义记忆 外部向量数据库 ✓ 持久 语义相似度搜索 用户偏好、结构化事实
程序记忆 System Prompt / 配置 ✓ 持久 按 ID / 全量加载 行为规则、工具列表

核心问题：无状态的代价

记忆类型分类 — 认知科学与工程的结合

将人类四类记忆映射到 Agent 系统，建立工程设计的理论基础。

为什么要分类？

不同记忆类型有不同的存储特性、访问模式和生命周期。分类是实现高效工程化的前提——知道什么该存在哪里、用什么方式检索。

工作记忆 → Context 情节记忆 → 时序 DB 语义记忆 → 向量 DB 程序记忆 → Config
不同场景的记忆选择
  • 个性化推荐 → 语义记忆（用户偏好）
  • 对话连续性 → 情节记忆（历史上下文）
  • 技能/规则更新 → 程序记忆（System Prompt）
  • 当前推理 → 工作记忆（Context Window）
  • 长期知识积累 → 语义记忆 + 整合 Pipeline
L2

Memory Manager 设计 — 六大职责

设计并实现统一管理记忆全生命周期的核心组件。

Memory Manager 的六大职责
1
Generation（生成） — 用 LLM 从对话/事件中提取结构化记忆片段（Memory Trace）
2
Storage（存储） — 将不同类型记忆写入对应后端（向量 DB 存语义，时序日志存情节）
3
Retrieval（检索） — 根据当前查询做语义相似度搜索，召回最相关历史记忆
4
Integration（集成） — 将检索到的记忆与当前状态、工具输出拼装成完整 Prompt
5
Updating（更新） — Agent 通过 Write-Back 主动修正/追加记忆，实现跨会话学习
6
Deletion（遗忘） — 淘汰过期低价值记忆，防止存储膨胀和噪声干扰
Memory Manager API 骨架
class MemoryManager: def load_user_context(self, user_id: str) -> dict: ... def retrieve(self, query: str, k: int = 5) -> list: ... def store(self, trace: MemoryTrace) -> None: ... def writeback(self, input: str, output: str) -> None: ... def consolidate(self) -> None: ... # 情节→语义 def forget(self, policy: str) -> None: ...

Oracle AI Database — 向量存储基础设施

搭建原生支持向量的 Oracle AI Database，创建记忆表，配置 HNSW 向量索引。

Oracle AI DB 核心特性
  • VECTOR 数据类型 — 原生存储高维嵌入向量
  • HNSW 向量索引 — 高效近似最近邻搜索（ANN）
  • COSINE 距离函数 — 语义相似度计算
  • SQL 原生集成 — 向量查询 + 结构化查询混合
  • JSON 列支持 — 存储记忆 Trace 的元数据
为什么选 Oracle AI DB？

无需单独部署向量数据库，在同一个数据库中同时处理：

  • 向量嵌入（语义记忆检索）
  • 结构化元数据（时间、用户ID、类型）
  • JSON 字段（灵活的记忆属性）
  • 事务保证（数据一致性）
建表 + 向量索引 SQL
-- 创建带向量列的语义记忆表 CREATE TABLE agent_memory ( id NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY, user_id VARCHAR2(100), content CLOB, embedding VECTOR(1536, FLOAT32), -- text-embedding-3-small memory_type VARCHAR2(20), -- semantic/episodic/procedural metadata JSON, created_at TIMESTAMP DEFAULT SYSTIMESTAMP ); -- HNSW 向量索引（近似最近邻，高效检索） CREATE VECTOR INDEX memory_vec_idx ON agent_memory (embedding) ORGANIZATION INMEMORY NEIGHBOR GRAPH DISTANCE COSINE WITH TARGET ACCURACY 95;

存储与语义检索 — 向量化写入与召回

实现记忆的向量化写入，以及基于语义相似度的高效召回。

LangChain × Oracle DB 集成
from langchain_community.vectorstores import OracleVS from langchain_openai import OpenAIEmbeddings import oracledb conn = oracledb.connect(user="agent_user", password="...", dsn="localhost/FREEPDB1") embeddings = OpenAIEmbeddings(model="text-embedding-3-small") vector_store = OracleVS.from_existing( connection=conn, embedding=embeddings, table_name="agent_memory" ) # 写入记忆 vector_store.add_texts(["用户喜欢简洁的回答风格"], metadatas=[{"type":"semantic"}]) # 语义检索 Top-5 results = vector_store.similarity_search("用户的回答偏好", k=5)
语义检索的关键参数
  • k（召回数量） — 通常 3–8，太多噪声大，太少漏信息
  • score_threshold — 设置相似度阈值过滤低质量结果
  • 距离函数 — COSINE（推荐，不受向量范数影响）
  • Embedding 模型 — text-embedding-3-small（平衡性能/成本）
L5

语义工具检索 — 解决工具集膨胀问题

用向量语义搜索动态加载工具，避免把所有工具塞进 Context Window。

⚠️
痛点：Agent 工具数量增长时，把所有工具描述塞入 Context Window 会快速消耗 Token 预算，超出上限后性能急剧下降。解决方案：只在运行时按需检索最相关的工具子集。

→

实现代码
class SemanticToolRetriever: def register(self, name, desc, fn): self.store.add_texts([desc], metadatas=[{"name":name,"tool":fn}]) def get_tools(self, task: str, k=5): docs = self.store.similarity_search(task, k=k) return [d.metadata["tool"] for d in docs]

记忆整合 — 情节记忆升华为语义记忆

将碎片化的历史事件记录提炼为通用知识，类比人类睡眠中的记忆巩固过程。

为什么需要整合？
  • 情节记忆是原始流水账，冗余高、难检索
  • 整合后的语义记忆更简洁、更通用
  • 减少存储空间，提升检索精准度
  • 去除矛盾和过时信息，保留最新共识
四步整合流程
①
聚类 — 按主题/实体对情节记忆做语义聚类
②
摘要 — LLM 将同主题多条记忆压缩提炼为一条
③
写入 — 整合后语义记忆向量化存入 Oracle DB
④
归档 — 原始情节记忆归档或按策略淘汰

Write-Back 回写循环 — Agent 自主学习的核心

让 Agent 在完成任务后主动将新知识写回记忆库，实现无监督自我进化。

Write-Back 闭环
Agent 执行任务
→
生成输出+反思
→
识别新知识
→
更新记忆库
→
下次启动加载
✓ 优势
  • Agent 自主学习，无需人工标注
  • 跨会话记忆逐步精化，减少重复错误
  • 支持用户偏好的动态自适应
  • 知识可在多 Agent 间共享传递
⚠ 注意事项
  • 设置置信度阈值，防止错误记忆污染
  • 高风险场景建议加人工审核门控
  • 提前设计记忆冲突解决策略
  • 遗忘机制必不可少，防无限膨胀

完整 Memory-Aware Agent — 所有模块组装

将 MemoryManager、SemanticToolRetriever、Extraction/Consolidation Pipeline 整合为完整的有状态 Agent。

完整 Agent 骨架代码
class MemoryAwareAgent: def __init__(self, llm, memory: MemoryManager, tools: SemanticToolRetriever): self.llm, self.memory, self.tools = llm, memory, tools def startup(self, user_id: str): # 1. 加载用户语义记忆 + 程序记忆（System Prompt） self.context = self.memory.load_user_context(user_id) self.system = self.memory.load_procedural(user_id) def chat(self, user_input: str) -> str: # 2. 语义检索相关记忆 memories = self.memory.retrieve(user_input, k=5) # 3. 语义检索相关工具 active_tools = self.tools.get_tools(user_input) # 4. 组装 Prompt prompt = self._build(user_input, memories, active_tools) # 5. LLM 推理 resp = self.llm.invoke(prompt) # 6. Write-Back 回写新知识 self.memory.writeback(user_input, resp.content) return resp.content
启动 vs 传统 Agent 的关键差异
阶段传统 Stateless AgentMemory-Aware Agent
启动空白 Prompt加载用户历史记忆 + 偏好
工具加载全量工具塞入 Context按需语义检索相关工具
推理无历史上下文携带相关历史记忆推理
会话结束全部遗忘自动提取新知识写回记忆库

术语表

Memory-Aware Agent

具备持久化记忆能力的 AI Agent，可跨会话积累知识并自主更新记忆库，与 Stateless Agent 相对。

Memory Trace

从对话/事件中提取的最小结构化记忆单元，包含事实内容、相关实体、记忆类型、置信度等字段。

Memory Manager

统一管理记忆全生命周期（生成/存储/检索/集成/更新/遗忘）的核心组件，是 Memory-Aware Agent 的中枢。

Write-Back Loop

Agent 完成任务后将新知识自动回写记忆库的闭环机制，实现无监督的跨会话持续学习。

Memory Consolidation

将情节记忆（原始事件）整合提炼为语义记忆（通用知识）的过程，类比人类睡眠时海马体的记忆巩固。

Semantic Tool Retrieval

基于语义相似度动态召回相关工具，避免将全部工具描述塞入 Context Window 导致 Token 膨胀。

HNSW 索引

Hierarchical Navigable Small World，高效的近似最近邻（ANN）向量搜索算法，Oracle AI DB 原生支持。

Oracle AI Database

Oracle 数据库的 AI 扩展，原生支持 VECTOR 数据类型和向量相似度查询，本课程的核心存储基础设施。

Embedding（嵌入向量）

将文本转化为高维数值向量的表示，语义相近的文本在向量空间中距离较近，是语义检索的数学基础。

Stateless vs Stateful

无状态 Agent 每次会话独立、重置；有状态 Agent 通过外部记忆库保持跨会话的连续性和学习能力。

LangChain

构建 LLM 应用的 Python 框架，本课程用于 Agent 编排、Prompt 管理和 Oracle AI DB 向量存储集成。

Context Window

LLM 单次推理能处理的最大 Token 数量，等同于 Agent 的"工作记忆"容量上限，是无状态化的根本原因。
